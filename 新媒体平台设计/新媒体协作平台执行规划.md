# 新媒体协作平台执行规划

## 1. 执行目标

本执行规划基于《新媒体协作平台完整方案》，用于指导学校社团新媒体协作平台从想法走向可落地的网站原型与第一版产品。

执行目标不是一次性替代秀米、微信公众号后台或微信群，而是先建立一个统一的协作中枢，让采编部、技术部、运维部在同一个平台里完成任务交接、文件提交、审核反馈和进度追踪。

第一阶段成功标准：

- 能创建一篇推文项目。
- 每篇推文项目都自动拆分为采编、封面、发布三个任务。
- 三个部门可以在同一个项目详情页中提交链接、文件和评论。
- 新媒体主任可以查看所有项目进度。
- 负责老师可以查看并审批重点内容。
- 运维部可以记录发布链接、发布时间和发布截图。

## 2. 总体推进节奏

建议按 5 个阶段推进。

| 阶段 | 周期建议 | 目标 | 主要交付物 |
| --- | --- | --- | --- |
| 阶段 1：需求确认 | 1-2 天 | 明确真实工作流和角色边界 | 需求确认表、角色清单、流程确认稿 |
| 阶段 2：信息架构与原型 | 2-4 天 | 画出平台页面和核心交互 | 页面结构图、低保真原型、字段清单 |
| 阶段 3：MVP 前端实现 | 5-10 天 | 做出可演示的静态前端 | React 页面、模拟数据、可运行 Demo |
| 阶段 4：试运行与修改 | 3-7 天 | 用真实推文流程测试平台 | 试运行记录、问题清单、改进版 |
| 阶段 5：后端与长期化 | 视情况推进 | 接入登录、数据库和通知 | 后端接口、数据库、权限系统 |

如果时间有限，应优先完成阶段 1-3，先形成一个可以演示和讨论的 MVP。

## 3. 阶段 1：需求确认

### 3.1 需要确认的问题

在正式设计页面前，需要向团队确认以下信息：

- 社团每月大约发布多少篇推文。
- 推文是否分栏目，例如活动宣传、通知公告、人物专访、赛事报道。
- 哪些推文必须由负责老师审批。
- 三个部门目前各有多少成员。
- 每个部门是否由负责人统一分配任务，还是干事可以自由领取任务。
- 秀米链接、封面文件、公众号预览链接目前如何传递。
- 公众号发布时间是否固定。
- 发布后是否需要记录阅读量、点赞量、转发量等数据。

### 3.2 输出物

需求确认阶段应产出一份简短表格。

建议字段：

- 角色名称。
- 人数。
- 当前主要工作。
- 平台中需要的权限。
- 最常遇到的问题。
- 希望平台解决的问题。

### 3.3 决策原则

第一版只解决高频问题，不做低频复杂功能。

优先级从高到低：

1. 任务流转是否清楚。
2. 文件和链接是否容易找到。
3. 责任人和截止时间是否明确。
4. 审批意见是否能留痕。
5. 数据复盘是否方便。

## 4. 阶段 2：信息架构与原型

### 4.1 页面优先级

第一版必须设计 5 个页面。

| 页面 | 目的 | 第一版重点 |
| --- | --- | --- |
| 工作台 | 用户登录后的总览 | 我的任务、待审批、即将发布、逾期提醒 |
| 项目看板 | 查看所有推文状态 | 按状态分组展示项目 |
| 项目详情页 | 单篇推文协作中心 | 三阶段任务、链接文件、评论、审批 |
| 部门空间 | 部门内部任务管理 | 采编、技术、运维分别查看任务 |
| 成员与权限 | 管理人员和角色 | 用户角色、所属部门、启用状态 |

第二版再设计：

- 选题库高级筛选。
- 通知中心独立页面。
- 数据复盘页面。
- 模板库页面。

### 4.2 核心页面布局

#### 工作台

工作台建议分为三列：

- 左侧：导航菜单。
- 中间：我的任务、我负责的项目。
- 右侧：待审批、即将发布、风险提醒。

卡片内容：

- 任务名称。
- 所属推文项目。
- 所属部门。
- 截止时间。
- 当前状态。
- 下一步操作按钮。

#### 项目看板

项目看板按状态分列：

- 待采编。
- 采编中。
- 待封面。
- 封面中。
- 待审批。
- 待发布。
- 已发布。
- 需返工。

每张项目卡片显示：

- 推文标题。
- 所属栏目。
- 预计发布时间。
- 当前负责人。
- 优先级。
- 是否逾期。

#### 项目详情页

项目详情页是 MVP 最重要页面。

必须包含：

- 项目基础信息。
- 三阶段流程条。
- 采编任务卡。
- 封面任务卡。
- 发布任务卡。
- 文件与链接列表。
- 评论与修改意见。
- 审批记录。

### 4.3 原型验收标准

原型完成后，应能回答以下问题：

- 新媒体主任如何创建一个项目。
- 采编部如何提交秀米链接。
- 技术部如何知道要做什么封面。
- 运维部如何拿到最终推文和封面。
- 负责老师如何审批重点推文。
- 如果被驳回，任务如何回到上一阶段。

## 5. 阶段 3：MVP 前端实现

### 5.1 技术路线

当前项目建议继续使用：

- Vite。
- React。
- TypeScript。
- lucide-react 图标。
- CSS Modules 或集中式 CSS。

第一版先使用前端模拟数据，不接后端。

这样可以快速验证页面结构和流程，不会被登录、数据库、接口权限拖慢。

### 5.2 建议目录结构

```text
src/
  App.tsx
  styles.css
  data/
    mockData.ts
  types/
    platform.ts
  components/
    AppShell.tsx
    StatusBadge.tsx
    RoleBadge.tsx
    TaskCard.tsx
    ProjectCard.tsx
    FileLinkList.tsx
    ApprovalPanel.tsx
  pages/
    Dashboard.tsx
    ProjectBoard.tsx
    ProjectDetail.tsx
    DepartmentSpace.tsx
    Members.tsx
```

如果为了快速演示，也可以先把页面和数据集中在 `App.tsx` 中，等页面稳定后再拆分组件。

### 5.3 类型设计

建议第一版至少定义以下 TypeScript 类型：

```ts
type Department = "采编部" | "技术部" | "运维部";

type Role =
  | "负责老师"
  | "新媒体主任"
  | "部门负责人"
  | "干事";

type ProjectStatus =
  | "草稿中"
  | "待采编"
  | "采编中"
  | "待封面"
  | "封面中"
  | "待审批"
  | "待发布"
  | "已发布"
  | "需返工"
  | "已归档";

type TaskType = "采编任务" | "封面任务" | "发布任务";

type TaskStatus =
  | "未开始"
  | "进行中"
  | "待审核"
  | "需修改"
  | "已完成"
  | "已逾期";
```

### 5.4 模拟数据

第一版建议准备：

- 1 位负责老师。
- 1 位新媒体主任。
- 3 位部门负责人。
- 6-9 位干事。
- 5 个推文项目。
- 每个项目 3 个任务。
- 每个项目至少 2 条文件或链接。
- 每个项目至少 2 条评论或审批记录。

示例项目：

- 迎新招新宣传推文。
- 社团活动预告推文。
- 人物专访推文。
- 活动回顾推文。
- 通知公告推文。

### 5.5 交互范围

MVP 前端可以实现轻量交互：

- 切换页面。
- 切换项目状态筛选。
- 点击项目卡片查看详情。
- 切换部门空间。
- 模拟审批通过或驳回。
- 模拟任务状态变化。

暂不实现：

- 真实登录。
- 真实文件上传。
- 真实消息推送。
- 真实数据库保存。
- 微信公众号接口。
- 秀米接口。

## 6. 阶段 4：试运行与修改

### 6.1 试运行方式

选择 1-2 篇真实推文，把它们完整录入平台 Demo。

测试流程：

1. 新媒体主任创建推文项目。
2. 采编部提交秀米链接。
3. 技术部根据推文提交封面。
4. 运维部记录公众号预览链接。
5. 负责老师或主任进行审批。
6. 运维部记录发布结果。
7. 团队成员反馈使用体验。

### 6.2 需要观察的问题

- 成员能否理解项目状态。
- 三个部门是否知道自己下一步要做什么。
- 评论区是否比微信群更容易追踪意见。
- 文件和链接是否容易找到。
- 负责人能否快速发现逾期任务。
- 页面信息是否过多或过少。

### 6.3 修改优先级

试运行后优先修改：

1. 影响主流程跑通的问题。
2. 角色权限看不清的问题。
3. 状态命名容易误解的问题。
4. 文件或链接找不到的问题。
5. 页面布局影响使用效率的问题。

## 7. 阶段 5：后端与长期化

当前端 Demo 被验证可用后，再进入后端设计。

### 7.1 后端能力

后端第一版需要：

- 用户登录。
- 角色权限。
- 项目增删改查。
- 任务增删改查。
- 文件链接记录。
- 评论记录。
- 审批记录。
- 发布数据记录。

### 7.2 数据库表

建议最小表结构：

- users。
- departments。
- roles。
- projects。
- tasks。
- attachments。
- comments。
- approvals。
- publish_metrics。

### 7.3 通知能力

通知第一版可以先做站内通知。

后续再考虑：

- 邮件提醒。
- 企业微信或飞书提醒。
- 微信群机器人提醒。

## 8. 任务分工建议

### 8.1 负责老师

- 确认哪些内容必须审批。
- 确认重要推文的质量标准。
- 参与试运行验收。

### 8.2 新媒体主任

- 提供真实推文流程。
- 确认页面模块和状态流转。
- 组织三个部门试运行。
- 收集反馈并决定优先级。

### 8.3 采编部

- 提供秀米链接使用习惯。
- 确认采编任务字段。
- 测试文案提交、修改、定稿流程。

### 8.4 技术部

- 提供封面尺寸和文件命名规范。
- 确认封面需求字段。
- 测试封面提交和返工流程。

### 8.5 运维部

- 提供公众号发布流程。
- 确认发布前检查清单。
- 测试预览链接、发布链接和截图记录流程。

### 8.6 开发执行者

- 根据方案实现页面和模拟数据。
- 保持页面风格清晰、低噪、适合日常使用。
- 每完成一个阶段进行构建检查。
- 保证后续可以逐步接入后端。

## 9. 里程碑与验收

### 9.1 里程碑一：方案确认

验收标准：

- 团队确认角色、部门和流程无重大遗漏。
- 推文状态和任务状态可以覆盖日常场景。
- 第一版范围被明确锁定。

### 9.2 里程碑二：原型确认

验收标准：

- 工作台、项目看板、项目详情页结构清楚。
- 三个部门能看懂自己的任务入口。
- 老师和主任能找到审批与总览入口。

### 9.3 里程碑三：前端 Demo

验收标准：

- 本地可以运行网站。
- 至少展示 5 个模拟推文项目。
- 每个项目都有采编、封面、发布三个任务。
- 可以查看项目详情。
- 可以查看文件链接、评论和审批记录。

### 9.4 里程碑四：真实流程试运行

验收标准：

- 至少 1 篇真实推文完整走完流程。
- 试运行问题被记录。
- 团队确认平台能减少重复沟通。

## 10. 风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 成员不愿使用新平台 | 仍然只在微信群沟通 | 第一版减少填写负担，只要求关键链接、状态和意见上平台 |
| 流程设计过重 | 每篇推文都要填太多信息 | 第一版字段保持最少，复杂字段放到第二版 |
| 权限太复杂 | 成员不知道自己能做什么 | 第一版角色不超过四层，部门负责人负责本部门审核 |
| 文件版本混乱 | 不知道哪个封面是最终版 | 文件区增加“定稿”标记和版本号 |
| 第三方接口不稳定 | 秀米或公众号无法直连 | 第一版只保存链接，不依赖接口 |
| 数据复盘没人填写 | 发布后数据缺失 | 运维发布任务中加入“数据待填写”提醒 |

## 11. 当前项目下一步建议

如果继续在当前 Vite + React 项目中实现网站，建议按以下顺序执行：

1. 重写现有乱码页面，改为新媒体协作平台工作台。
2. 新建模拟数据，覆盖角色、部门、项目、任务、文件、评论和审批。
3. 实现单页多视图切换：工作台、项目看板、项目详情、部门空间、成员管理。
4. 使用状态标签、任务卡片、项目卡片和审批面板组织界面。
5. 运行 `npm.cmd run build` 验证构建。
6. 启动本地开发服务器，让团队查看 Demo。

第一版不需要删除任何现有文件。若后续确实需要清理旧文件，应逐个明确路径删除，不进行批量删除。
