# 新媒体协作平台完整方案

## 1. 项目定位

本平台面向学校社团新媒体团队，目标是把“选题、推文、封面、发布、复盘”整合到一个可追踪、可协作、可沉淀的平台中。

当前三个部门的工作链路较长：

- 采编部负责使用秀米制作推文。
- 技术部根据推文内容制作封面。
- 运维部将推文和封面整理后发送到微信公众号后台。

如果主要依赖微信群沟通，容易出现信息分散、版本混乱、责任不清、截止时间无人跟进等问题。平台应帮助三个部门把协作流程固定为明确的任务流，让负责老师、新媒体主任、部门负责人和干事都能快速了解“谁在做、做到哪一步、还差什么、什么时候发布”。

## 2. 使用对象与角色权限

### 2.1 负责老师

负责老师主要关注整体方向、重点内容质量和最终风险把关。

核心权限：

- 查看所有推文项目、发布排期和整体进度。
- 对重点推文进行审批、驳回或提出修改意见。
- 查看每个项目的历史修改记录和负责人。
- 查看发布后的基础数据复盘。

典型使用场景：

- 在活动前查看宣传推文是否按计划推进。
- 对涉及学校形象、重要活动、正式通知的推文进行最终确认。
- 在复盘阶段查看近期推文传播效果。

### 2.2 新媒体主任

新媒体主任是平台的主要统筹者，负责把需求转成任务并推动各部门协作。

核心权限：

- 创建选题和推文项目。
- 指定栏目、优先级、发布时间和项目负责人。
- 给采编部、技术部、运维部分配任务。
- 调整排期、处理延期、协调返工。
- 查看所有部门的任务负载和完成情况。

典型使用场景：

- 将社团活动宣传需求录入为一个新项目。
- 给采编负责人分配文案任务，给技术负责人分配封面任务。
- 在项目卡住时查看是哪一环节未完成，并进行催办。

### 2.3 部门负责人

部门负责人负责管理本部门任务，确保本部门交付物质量达标。

采编部负责人：

- 接收推文项目中的采编任务。
- 分配干事撰写、编辑或校对推文。
- 审核秀米链接、文案内容和排版质量。
- 将采编任务提交给技术部或主任审核。

技术部负责人：

- 查看采编部提供的推文主题、标题、内容重点和封面需求。
- 分配干事制作封面。
- 审核封面尺寸、风格、文字准确性和导出文件。
- 将封面定稿提交给运维部。

运维部负责人：

- 接收定稿推文和封面。
- 分配干事进行公众号后台排版、预览和发布。
- 确认发布时间、发布截图和发布链接。
- 录入发布后的基础数据。

### 2.4 各部门干事

干事是具体任务执行者。

核心权限：

- 查看分配给自己的任务。
- 领取任务、更新进度、提交作品。
- 上传秀米链接、封面文件、公众号预览链接或发布截图。
- 在任务评论区回复修改意见。
- 提交“已完成”“需要协助”“无法按时完成”等状态。

典型使用场景：

- 采编干事上传秀米初稿链接。
- 技术干事上传封面第一版图片。
- 运维干事上传公众号发布截图和推文链接。

## 3. 组织结构设计

平台默认组织层级如下：

```text
负责老师
└── 新媒体主任
    ├── 采编部负责人
    │   └── 采编部干事
    ├── 技术部负责人
    │   └── 技术部干事
    └── 运维部负责人
        └── 运维部干事
```

后续可扩展：

- 临时专项组：例如迎新专题、毕业季专题、校运会专题。
- 栏目组：例如活动宣传、人物专访、赛事报道、通知公告。
- 审核组：用于重要内容的二次审核。

## 4. 核心业务流程

### 4.1 选题立项

新媒体主任或授权负责人创建推文项目。

必填信息：

- 推文标题或暂定标题。
- 所属栏目或活动。
- 内容主题。
- 预计发布时间。
- 优先级：普通、重要、紧急。
- 是否需要负责老师审批。
- 采编负责人、技术负责人、运维负责人。

输出结果：

- 生成一个推文项目。
- 自动生成采编任务、封面任务、发布任务。
- 项目进入“待采编”状态。

### 4.2 采编制作

采编部根据选题完成推文内容。

主要动作：

- 领取采编任务。
- 填写采访对象、素材来源、文案方向。
- 上传秀米链接。
- 标记当前版本：初稿、修改稿、定稿。
- 提交给采编负责人审核。

完成标准：

- 秀米链接可正常打开。
- 标题、正文、图片、排版基本完成。
- 已确认无明显错别字和事实错误。
- 如需老师审批，已进入审批流程。

### 4.3 技术制图

技术部根据推文内容制作封面。

主要动作：

- 查看推文主题、标题、视觉关键词和封面用途。
- 上传封面初稿。
- 根据意见修改。
- 提交定稿文件。

建议规范：

- 微信公众号封面首图：常用比例 2.35:1，可按团队实际规范固定尺寸。
- 次图或分享图：根据公众号后台要求另设模板。
- 文件命名建议：日期_栏目_推文标题_封面版本。

完成标准：

- 封面文字无错字。
- 主视觉与推文主题一致。
- 尺寸符合公众号发布要求。
- 已上传最终图片文件。

### 4.4 运维发布

运维部负责公众号后台整理和发布。

主要动作：

- 获取定稿秀米链接和封面文件。
- 在公众号后台导入或排版。
- 生成预览并提交确认。
- 按计划时间发布。
- 上传发布截图和正式推文链接。

完成标准：

- 公众号后台内容与定稿一致。
- 封面、标题、摘要、作者等信息正确。
- 已按计划发布或注明延迟原因。
- 已保存发布证明。

### 4.5 审批与返工

审批可由负责老师、新媒体主任或部门负责人完成。

审批结果：

- 通过：进入下一阶段。
- 驳回：回到上一阶段并记录修改意见。
- 暂缓：保留任务但暂停发布。

返工要求：

- 每次返工必须填写原因。
- 返工意见应绑定到具体任务或文件。
- 返工后重新提交审核，形成记录闭环。

## 5. 状态设计

推文项目状态建议如下：

| 状态 | 含义 |
| --- | --- |
| 草稿中 | 选题尚未正式分配 |
| 待采编 | 项目已创建，等待采编部开始 |
| 采编中 | 采编部正在制作推文 |
| 待封面 | 推文内容已基本确认，等待技术部制作封面 |
| 封面中 | 技术部正在制作封面 |
| 待审批 | 推文或封面等待老师、主任或负责人审核 |
| 待发布 | 内容和封面已定稿，等待运维部发布 |
| 已发布 | 已在公众号正式发布 |
| 需返工 | 当前阶段被驳回，需要修改 |
| 已归档 | 项目结束并完成复盘 |

任务状态建议如下：

| 状态 | 含义 |
| --- | --- |
| 未开始 | 已分配但尚未处理 |
| 进行中 | 执行者正在处理 |
| 待审核 | 已提交，等待负责人审核 |
| 需修改 | 审核未通过，需要返工 |
| 已完成 | 当前任务完成 |
| 已逾期 | 超过截止时间仍未完成 |

## 6. 页面模块设计

### 6.1 工作台

工作台是登录后的首页，帮助用户快速知道今天要做什么。

核心内容：

- 我的今日任务。
- 我负责的项目。
- 待我审批。
- 即将发布。
- 已逾期任务。
- 最新评论和提醒。

不同角色看到的重点不同：

- 负责老师看到审批和整体进度。
- 新媒体主任看到所有项目、部门负载和风险提醒。
- 部门负责人看到本部门任务与待审核内容。
- 干事看到自己的任务清单。

### 6.2 选题库

选题库用于管理所有推文项目。

筛选条件：

- 月份。
- 栏目。
- 部门。
- 状态。
- 发布时间。
- 负责人。
- 优先级。

常用视图：

- 日历视图：按发布时间查看排期。
- 看板视图：按状态查看项目流转。
- 列表视图：适合批量检索和归档。

### 6.3 项目详情页

项目详情页是单篇推文的协作中心。

主要区域：

- 基础信息：标题、栏目、发布时间、优先级、负责人。
- 流程看板：采编、封面、发布三个阶段。
- 文件与链接：秀米链接、封面文件、公众号预览、发布链接。
- 评论区：集中讨论问题，支持 @ 成员。
- 审批记录：记录谁在什么时候通过或驳回。
- 修改历史：记录关键版本变化。

### 6.4 部门空间

三个部门分别拥有自己的工作空间。

采编部空间：

- 待写推文。
- 待校对推文。
- 被驳回推文。
- 已定稿推文。

技术部空间：

- 待制作封面。
- 待修改封面。
- 已定稿封面。
- 封面模板库。

运维部空间：

- 待发布推文。
- 待确认预览。
- 已发布推文。
- 发布数据待填写。

### 6.5 文件与链接区

文件与链接区用于减少“文件在谁那里”的问题。

支持内容：

- 秀米编辑链接。
- 秀米预览链接。
- 封面图片。
- 公众号后台预览链接。
- 发布截图。
- 正式推文链接。
- 活动素材包链接。

建议规则：

- 每个文件必须标注上传者、上传时间和用途。
- 定稿文件应明确标记为“最终版”。
- 旧版本保留但不作为默认使用版本。

### 6.6 通知中心

通知中心负责把协作变化及时传达给相关人员。

通知类型：

- 任务被分配。
- 截止时间临近。
- 有人 @ 我。
- 审批通过或驳回。
- 项目状态变化。
- 发布任务即将开始。
- 任务已逾期。

### 6.7 数据复盘

数据复盘用于帮助社团积累经验。

基础指标：

- 阅读量。
- 点赞量。
- 在看量。
- 转发量。
- 收藏量。
- 推文完成周期。
- 返工次数。

复盘维度：

- 哪类栏目传播效果更好。
- 哪个阶段最容易延期。
- 哪些推文返工次数较多。
- 哪些模板或封面风格表现更稳定。

## 9. MVP 版本规划

### 9.1 第一版：基础协作可用

第一版目标是让团队真正把日常推文流转搬到平台上。

必须实现：

- 账号与角色。
- 推文项目创建。
- 项目看板。
- 采编、封面、发布三阶段任务。
- 任务状态流转。
- 评论与 @ 提醒。
- 秀米链接、封面文件、公众号链接管理。
- 基础审批记录。

暂不实现：

- 微信公众号接口直连。
- 秀米接口直连。
- 自动生成封面。
- 复杂数据分析。

### 9.2 第二版：管理效率提升

第二版目标是让主任和负责人更容易管理团队。

建议实现：

- 数据复盘看板。
- 封面模板库。
- 推文模板库。
- 自动截止提醒。
- 移动端适配优化。
- 按栏目生成月度统计。

### 9.3 第三版：自动化与沉淀

第三版目标是减少重复劳动，并形成社团自己的内容资产。

可探索：

- 微信公众号接口集成。
- 秀米深度集成。
- 自动生成周报。
- 内容素材库。
- 历史优秀案例库。
- 根据数据推荐发布时间和栏目方向。

## 10. 网站信息架构

建议导航结构：

```text
工作台
选题库
项目看板
部门空间
  ├── 采编部
  ├── 技术部
  └── 运维部
文件与链接
通知中心
数据复盘
成员管理
系统设置
```

首屏建议显示：

- 左侧：主导航。
- 中间：我的任务和项目看板。
- 右侧：通知、审批、即将发布。

整体设计应更像“编辑部工作台”，而不是宣传型官网。重点是清晰、低噪、信息密度适中、适合频繁使用。

## 11. 推荐视觉风格

设计关键词：

- 校园媒体。
- 编辑部。
- 清晰有序。
- 轻量协作。
- 稳定可信。

建议风格：

- 背景使用浅灰、米白或低饱和蓝灰。
- 关键状态使用明确色彩区分，例如绿色代表已完成，橙色代表待审核，红色代表逾期。
- 卡片圆角保持克制，不做过度装饰。
- 页面信息应方便扫视，避免大面积营销文案。
- 图标用于任务、审批、通知、文件、数据等功能入口。

## 12. 后续前端实现建议

当前项目可继续使用 Vite + React + TypeScript 实现。

建议第一阶段页面：

- `Dashboard`：工作台。
- `ProjectBoard`：项目看板。
- `ProjectDetail`：项目详情页。
- `DepartmentSpace`：部门空间。
- `Notifications`：通知中心。

建议组件：

- `StatusBadge`：状态标签。
- `RoleBadge`：角色标签。
- `TaskCard`：任务卡片。
- `ProjectCard`：项目卡片。
- `Timeline`：项目流程时间线。
- `FileLinkList`：文件与链接列表。
- `ApprovalPanel`：审批面板。

建议数据先使用前端静态 mock：

- 先模拟 3 个部门、8 个用户、5 个推文项目。
- 等交互和流程稳定后，再接入后端数据库。

需要注意：

- 现有 `src/App.tsx` 中中文内容存在乱码，后续实现网站时建议直接重写页面文本。
- 不建议在第一版做复杂第三方接口集成，先把内部流程跑通。
- 平台不直接替代秀米和微信公众号后台，而是作为任务、链接、文件、审批和复盘的统一协作层。

## 13. 验收标准

方案文档层面的验收标准：

- 已明确平台目标和适用对象。
- 已覆盖负责老师、新媒体主任、部门负责人、干事四类角色。
- 已覆盖采编部、技术部、运维部三个部门的工作流程。
- 已设计推文项目从选题到发布的完整状态流转。
- 已列出页面模块、数据对象、权限矩阵和 MVP 优先级。
- 已给出后续前端实现方向。

第一版产品层面的验收标准：

- 新媒体主任可以创建一个推文项目。
- 项目自动生成采编、封面、发布三个任务。
- 三个部门成员可以围绕同一项目提交链接、文件和评论。
- 部门负责人可以审核本部门任务。
- 负责老师可以查看并审批重点推文。
- 运维部可以上传发布链接和发布截图。
- 所有人都能清楚看到项目当前状态和下一步责任人。
